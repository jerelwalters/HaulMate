# Idempotent Sync Contract

This contract supports the P0 durable outbox. The app writes locally first,
then retries one logical operation until Supabase returns a stable server
result.

## Load Snapshot RPC

Call `public.apply_load_sync_operation` through Supabase RPC as an
authenticated user.

Parameters:

| Name | Type | Required | Notes |
|---|---|---:|---|
| `operation_id` | UUID | Yes | Client-generated outbox operation ID. |
| `idempotency_key` | Text | Yes | Stable key for exactly one logical operation. Max 160 characters. |
| `load` | JSON object | Yes | Full load snapshot. Required fields: `id`, `customer_id`, `reference_number`, `status`, `line_haul_rate`, `loaded_miles`. Optional financial and reconciliation fields may be included. |
| `expected_server_updated_at` | Timestamp | No | Last server `updated_at` seen by the client. Use this for updates that must not overwrite newer server state. |

The `load.id` value is generated by the client and becomes
`public.loads.id`. The server fills `user_id` from `auth.uid()` and never
trusts a user ID from the payload.

## Response Shape

The RPC returns JSON:

```json
{
  "operation_id": "00000000-0000-0000-0000-000000000901",
  "idempotency_key": "load:abc:v1",
  "operation_type": "load.upsert",
  "target_table": "loads",
  "target_id": "00000000-0000-0000-0000-000000000691",
  "result": "applied",
  "error_code": null,
  "reconciliation": {
    "server_created_at": "2026-07-01T06:22:49Z",
    "server_updated_at": "2026-07-01T06:22:49Z",
    "server_status": "accepted"
  }
}
```

Retries with the same idempotency key and identical payload return the stored
response unchanged. Reusing an idempotency key with a different payload fails
with SQLSTATE `22023`.

## Rejection Codes

| Code | Meaning | Client action |
|---|---|---|
| `stale_server_record` | `expected_server_updated_at` does not match the current server row. | Fetch the row and reconcile. |
| `missing_server_record` | The client expected an existing row, but the server has none. | Fetch or recreate through explicit user action. |
| `invalid_status_transition` | The requested load status moves outside the allowed P0 state path. | Keep the local operation failed and ask for review. |
| `financial_conflict` | The operation changes financial fields after the load is `invoiced` or `paid`. | Require explicit user review before creating a revision/follow-up operation. |

Rejected operations are also stored in `public.sync_operations`, so retries
return the same rejection without partially changing the load.

## Security Rules

- `sync_operations` is a server replay ledger. Authenticated clients can read
  only their own rows, but cannot insert, update, or delete ledger rows.
- `apply_load_sync_operation` is the only authenticated write path for the
  replay ledger.
- The RPC writes only rows owned by `auth.uid()` and relies on the existing
  same-user foreign keys for customer and vehicle ownership.
